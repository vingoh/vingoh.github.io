## 前言

之前写agent demo的第一步是从封装LLM开始的，那这次写框架也还是从这出发。一部分沿用了之前的内容，包括对think/llm_client封装，主要的修改则在于将think函数的返回类型从string改为了统一的格式，同时对于工具的调用接口也做了类似的修改。

> 不知为何，当年手搓back propogation的感觉又回来了

## 主要解决的问题

旧实现主要关注流式文本拼接，返回值是纯字符串。这个模式在简单对话中够用，但在以下场景会遇到限制：

- 需要读取`tool_calls`时，缺少统一字段。
- 需要统计`usage`时，调用方只能自行补逻辑。
- 切换`stream=True/False`时，返回类型不一致，调用分支复杂。

本次改动后，上述信息都被收敛进`LLMResponse`，调用方可以用同一套代码处理不同模式。

## 核心结构变化

### 新增统一响应结构

新增了三个`Pydantic`模型：

- `ToolCall`：统一工具调用结构，包含`name`、`args`、`call_id`。
- `UsageInfo`：统一token统计结构，包含`prompt_tokens`、`completion_tokens`、`total_tokens`。
- `LLMResponse`：统一最终返回结构，包含`text`、`tool_calls`、`finish_reason`、`usage`、`raw`。

> 很奇怪，这种我会习惯叫作数据结构的东西这里都叫模型，好不适应，后面还是叫数据结构吧

### think接口变化

主要变化有以下几点：

- 新增参数：
    - `stream: bool`，流式输出开关
    - `tools: Optional[List[Dict[str, Any]]] = None`，结构化的工具调用输入
    - `tool_choice: Optional[Any] = None`，工具开关
- 返回类型：`LLMResponse`，由string改为结构化输出

这里入参的变化主要是基于兼容OpenAI格式的输入进行设计的。一个常见的OpenAI完整请求大概长这样：
```json
curl https://api.openai.com/v1/chat/completions \
-H "Content-Type: application/json" \
-H "Authorization: Bearer $OPENAI_API_KEY" \
-d '{
  "model": "gpt-4-turbo",
  "messages": [
    {
      "role": "user",
      "content": "What'\''s the weather like in Boston today?"     <==用户的问题
    }
  ],
  "tools": [                   <===用户提供工具集让模型来选择使用哪个可以解决用户的问题
    {
      "type": "function",
      "function": {
        "name": "get_current_weather",
        "description": "Get the current weather in a given location",
        "parameters": {
          "type": "object",
          "properties": {
            "location": {
              "type": "string",
              "description": "The city and state, e.g. San Francisco, CA"
            },
            "unit": {
              "type": "string",
              "enum": ["celsius", "fahrenheit"]
            }
          },
          "required": ["location"]
        }
      }
    }
  ],
  "tool_choice": "auto"
}'
```
可以看到model和messages的部分是之前都已经实现了的，所以这次新增了tools和tool_choice的输入，通过格式化输入的方式而非直接将可用工具和content混在一起，提高模型调用的可控性，降低后续融入tool调用的复杂性。

```python
class LLMResponse(BaseModel):
    """统一LLM响应结构（流式/非流式）"""

    text: str = ""
    tool_calls: List[ToolCall] = Field(default_factory=list)
    finish_reason: Optional[str] = None
    usage: Optional[UsageInfo] = None
    raw: Optional[Dict[str, Any]] = None
```

类似的，一个OpenAI格式的输出就会长这样：

```json
{
  "id": "chatcmpl-abc123",
  "object": "chat.completion",
  "created": 1699896916,
  "model": "gpt-3.5-turbo-0125",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": null,
        "tool_calls": [              ===>大模型根据用户的问题和用户提供的工具集返回了可能有用的函数
          {
            "id": "call_abc123",
            "type": "function",
            "function": {
              "name": "get_current_weather",  ===>用户可以在调用了该函数后在下一轮对话把该函数调用结果反馈给大模型，然后就可以得到一个最终的答复
              "arguments": "{\n\"location\": \"Boston, MA\"\n}"
            }
          }
        ]
      },
      "logprobs": null,
      "finish_reason": "tool_calls"
    }
  ],
  "usage": {
    "prompt_tokens": 82,
    "completion_tokens": 17,
    "total_tokens": 99
  }
}
```

之前在agent demo中我们获取工具调用的方法是在prompt中让模型以固定格式进行返回，但这样做稳定性较差，且依赖模型本身的理解、指令遵循能力，在不同模型上的表现差异较大。而通过直接使用模型格式化的返回，我们可以直接从中解析出下一步需要调用的工具和参数，也就是这次新定义的ToolCall结构：
```python
class ToolCall(BaseModel):
    """统一工具调用结构"""

    name: str = ""
    args: str = ""
    call_id: str = ""
```

此外，在后面还可以看到一个usage部分，通过这里可以看到此次token的用量信息，因此也新增了一个相应的结构用来统计，方便后续对比调试：
```python
class UsageInfo(BaseModel):
    """统一token使用量结构"""

    prompt_tokens: Optional[int] = None
    completion_tokens: Optional[int] = None
    total_tokens: Optional[int] = None
```

### 输出解析

因为返回值不再是简单的字符串而是结构化的LLMResponse，因此需要在拿到llm的输出结果后进行解析构造，新增了两个内部解析方法：

- `_parse_stream_response()`：负责流式分片解析。
- `_parse_non_stream_response()`：负责非流式响应解析。

在流式输出时：

- 按chunk累计`delta.content`并实时打印。
- 捕获`finish_reason`。
- 处理`delta.tool_calls`分片，把同一`index`上的`arguments`逐段拼接。
- 最终把文本、工具调用、结束原因、usage统一封装为`LLMResponse`返回。

这里比较重要的一点是对tools相关的解析，因为在流式模式里，delta.tool_calls的 arguments可能是分段返回的。比如：

```json
chunk1.delta.tool_calls = [
  {
    "index": 0,
    "id": "call_abc",
    "function": {
      "name": "Search",
      "arguments": "{\"query\":\"英伟"
    }
  }
]

chunk2.delta.tool_calls = [
  {
    "index": 0,
    "function": {
      "name": "",
      "arguments": "达最新GPU "
    }
  }
]
```
这种情况下同一个tool的参数被截短在两个chunks中返回。如果是content这种纯字符串，可以直接拼接，但是tools需要tool与args匹配，而匹配则是通过tool的index进行维护。

```python
delta_tool_calls = getattr(delta, "tool_calls", None) or []
for tc in delta_tool_calls:
    index = getattr(tc, "index", 0)
    entry = tool_call_chunks.setdefault(index, {"name": "", "args": "", "call_id": ""})
    call_id = getattr(tc, "id", None)
    if call_id:
        entry["call_id"] = call_id

    function = getattr(tc, "function", None)
    if function:
        name_part = getattr(function, "name", "") or ""
        args_part = getattr(function, "arguments", "") or ""
        if name_part:
            entry["name"] = name_part
        if args_part:
            entry["args"] += args_part
tool_calls = [
    ToolCall(name=item["name"], args=item["args"], call_id=item["call_id"])
    for _, item in sorted(tool_call_chunks.items(), key=lambda pair: pair[0])
]
```

## Observability

在返回的结构中除了一众“有用”的数据之外，把原始的LLM返回也通过dump的方式存在字典中返回，这更多是考虑后续的可调试性和观测性，在正式功能之后可能会作为一个可选项进行返回。
