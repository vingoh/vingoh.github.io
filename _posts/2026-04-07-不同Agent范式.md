## ReAct

### ReAct结构

流程的核心是一个thought-action-observation的循环，每一步的思考$th_t$和行动$a_t$基于之前所有的行动和观察以及初始问题$q$:

$$ \left(th_t,a_t\right)=\pi\left(q,(a_1,o_1),\ldots,(a_{t-1},o_{t-1})\right) $$

随后根据行动$a_t$中的指示调用工具，工具返回新的观察$o_t$，加入到下一个循环的上下文。

### 工具

工具是所有agent的重要组成部分，负责与外界的交互，包括action和observation都需要依赖tool完成。通常来说，工具为"老式编程"的产物，为api或者某些封装好的代码。

对于LLM来说，重点则在于要知道有哪些tool可供选择、每个tool的作用以及如何调用每个tool，这也分别对应着一个tool定义中需要的三个属性：`name, description, execution logic`。代码中具体的：

```python!
search_description = "一个网页搜索引擎。当你需要回答关于时事、事实以及在你的知识库中找不到的信息时，应使用此工具。"
toolExecutor.registerTool("Search", search_description, search)
```

如上所见，通常还需要一个`ToolExecutor`类负责工具的管理，通常需要实现对的注册、获取可用工具以及获取具体的可执行函数。


### Agent
接下来看一下一个Agent通常需要的几个重要模块。

#### System Prompt

提示词是LLM的输入，system prompt则是用户不显示感知情况下，提前设定好，与用户给出的提示词一同输入给LLM的“隐藏”提示词。文中给出一个提示词范例如下：

```python!
REACT_PROMPT_TEMPLATE = """
请注意，你是一个有能力调用外部工具的智能助手。

可用工具如下:
{tools}

请严格按照以下格式进行回应:

Thought: 你的思考过程，用于分析问题、拆解任务和规划下一步行动。
Action: 你决定采取的行动，必须是以下格式之一:
- `{{tool_name}}[{{tool_input}}]`:调用一个可用工具。
- `Finish[最终答案]`:当你认为已经获得最终答案时。
- 当你收集到足够的信息，能够回答用户的最终问题时，你必须在Action:字段后使用 Finish[最终答案] 来输出最终答案。

现在，请开始解决以下问题:
Question: {question}
History: {history}
"""
```

<div class="note insight">

💡 prompt通常需要具有以下特点
* 具有动态上下文，如上面的`{question} {history}`，在不同迭代中动态插入对话历史、执行结果等信息
* 给出可用工具列表，让模型
* **严格规定输出格式**，这一步是LLM嵌入Agent框架的关键，输出格式的错误将导致工具的调用解析问题，以及循环无法提前终止
</div>


#### 核心循环

一个ReAct的核心是一个循环，用伪代码可以简单总结为如下的流程：



```
while:
    组装提示词，将tools、question、history插入到system prompt中
    调用llm思考
    解析输出结果，分离thought和action
    进一步从action分理出tool和所需参数
    调用工具，得到observation
    action和observation加入到history中
    
```

#### ReAct特点

ReAct Agent的优点有：
* **高可解释性**，如果想，每一步通过`thought`可以清晰看到模型的想法，下一步要调用的工具等等
* **动态规划与纠错能力**，通过`observation`动态更新下一步的想法和动作，在一定程度上可以对之前的错误进行纠偏
* **工具协同能力**，这个比较好理解，调用Tool解决具体问题

<div class="note insight">

💡 我的想法
这里虽然提到ReAct结构有一定的纠偏能力，但我认为这只能执行一些短期的纠偏，因为其step by step的架构使得其无法回看也无法执行长期的规划，有可能陷入局部最优
</div>

ReAct Agent的缺点有：
* **对LLM自身能力的强依赖**。一方面，模型的推理能力在很大程度上影响`thought`的输出质量，能力弱的模型可能做出错误的规划。另一方面，如上一期中所提到的，LLM输出的`thought`是“软硬编码”交界处，如果LLM的指令遵循能力不足以按照指示的规范进行输出，会导致工具无法调用、流程中断。


<div class="note insight">

💡 我的想法
`history`会不断累积增加，如果不做处理会使得上下文过长，可能需要一些处理方法

</div>

## Plan-and-Solve

### 结构
ReAct架构中在每个迭代中将thought与action融和在一起，再用当前迭代的observation结合之前迭代的一起融和为thought进行下一个迭代。而Plan-and-Solve则将thought和action分离开来，改为两个相对独立的阶段planning和solving。

Planning阶段不解决问题也不执行任何action，只根据问题$q$制定计划：

$$P = \pi_{\text{plan}}(q)$$ 


Solving极端严格按照计划执行，每一步$s_i$取决于问题$q$，计划$P$以及前面的执行结果:

$$s_i = \pi_{\text{solve}}(q, P, (s_1, \dots, s_{i-1}))$$

最后一次的执行结果$s_n$即为最终的结果。

### 规划阶段

文中通过一个不调用工具的纯提示词优化，来展示Plan-and-Solve在推理方面的优势。给出的示例任务为：
> 一个水果店周一卖出了15个苹果。周二卖出的苹果数量是周一的两倍。周三卖出的数量比周二少了5个。请问这三天总共卖出了多少个苹果？

可以看出这是一个非常明显的可以“一步一步”解决的问题，是一个非常适合planning的场景。

而对于planning的prompt也给出了对应的示例：
````python
PLANNER_PROMPT_TEMPLATE = """
你是一个顶级的AI规划专家。你的任务是将用户提出的复杂问题分解成一个由多个简单步骤组成的行动计划。
请确保计划中的每个步骤都是一个独立的、可执行的子任务，并且严格按照逻辑顺序排列。
你的输出必须是一个Python列表，其中每个元素都是一个描述子任务的字符串。

问题: {question}

请严格按照以下格式输出你的计划,```python与```作为前后缀是必要的:
```python
["步骤1", "步骤2", "步骤3", ...]
```
"""
````

可以看到，与ReAct使用的prompt在一些方面是类似的，如规定了LLM的角色，有着可动态插入的内容，规定了输出的格式。不同的地方在于任务描述为输出一个具体的计划，以及输出格式要求按照step输出计划。进一步将次prompt封装在一个planner类中，此类只通过用户输入和以上prompt输出对应的plan。

### 执行

在planner完成任务规划后，需要通过执行器executor来完成规划中的每一项人物。执行器实际上有两个任务：按照计划执行任务，以及记录任务执行后的状态。因此可以分析执行器需要的内容有：原始问题，完整的计划，当前的步骤，之前的每一步的步骤和执行结果。提示词的示例如下：

```python
EXECUTOR_PROMPT_TEMPLATE = """
你是一位顶级的AI执行专家。你的任务是严格按照给定的计划，一步步地解决问题。
你将收到原始问题、完整的计划、以及到目前为止已经完成的步骤和结果。
请你专注于解决“当前步骤”，并仅输出该步骤的最终答案，不要输出任何额外的解释或对话。

# 原始问题:
{question}

# 完整计划:
{plan}

# 历史步骤与结果:
{history}

# 当前步骤:
{current_step}

请仅输出针对“当前步骤”的回答:
"""

```

## Reflection

### 结构

在之前的结构中，一旦任务完成，整个执行流程即结束，无论结果是否存在瑕疵或者流程有一些弯路仍有可改进之处。而reflection结构中引入了执行后的“审查”机制，通过在任务完成后反思整个流程，来对全流程进行优化。个人认为某种程度上来说，相当于一个hyper react。一个reflection结构的agent通常会有如下流程：
1. 执行execution 使用React或Plan-Solve初次尝试
2. 反思reflection 调用独立或特殊提示词的大模型，评价前一步执行的结果
3. 优化refinement 根据反馈优化初次尝试的结果，生成更完美的方案

公式化的来说，假设$O_i$是第$i$次迭代产生的输出（$O_0$为初始输出），反思模型 $\pi_{\text{reflect}}$会生成针对$O_i$的反馈$F_i$：

$$
F_i = \pi_{\text{reflect}}(\text{Task}, O_i)
$$

随后，优化模型$\pi_{\text{refine}}$会结合原始任务、上一版输出以及反馈，生成新一版的输出 $O_{i+1}$：
$$
O_{i+1} = \pi_{\text{refine}}(\text{Task}, O_i, F_i)
$$

### 记忆模块

虽然在前面介绍的两种范式种，也会将过往的执行结果加入到后续上下文中，也是某种程度上的记忆，但reflection中将每次反思与优化的结果记录下来作为记忆，显示提供给后续的反思模块进行分析。

<div class="note insight">

💡 我的想法
relfection看起来有点像一个高维度的ReAct，同样是通过观察来获得反馈，再优化动作，但二者仍有明显区别。一方面，react只对下一个单一动作做优化，而reflection优化的是整个行动链条，相当于在更高一级上做优化。另一方面，虽然二者都将历史记录输入给了LLM，但react只是隐式的融入上下文，LLM不一定能真的从中获取经验；而reflection显式的通过一个独立LLM分析过往trajectory，能更多更好的从中获取经验，单个trajectory中获取短期记忆，多个trajectory之间获取长期记忆。
</div>

### 任务范例

对于reflection，给出的示例任务是：
> 编写一个Python函数，找出1到n之间所有的素数 (prime numbers)。


Reflection需要对初始执行、反思、优化三步分别设计提示词，给出的范例如下：

````python
#=================初次执行=================
INITIAL_PROMPT_TEMPLATE = """
你是一位资深的Python程序员。请根据以下要求，编写一个Python函数。
你的代码必须包含完整的函数签名、文档字符串，并遵循PEP 8编码规范。

要求: {task}

请直接输出代码，不要包含任何额外的解释。
"""

#=================反思=================
REFLECT_PROMPT_TEMPLATE = """
你是一位极其严格的代码评审专家和资深算法工程师，对代码的性能有极致的要求。
你的任务是审查以下Python代码，并专注于找出其在<strong>算法效率</strong>上的主要瓶颈。

# 原始任务:
{task}

# 待审查的代码:
```python
{code}
```

请分析该代码的时间复杂度，并思考是否存在一种<strong>算法上更优</strong>的解决方案来显著提升性能。
如果存在，请清晰地指出当前算法的不足，并提出具体的、可行的改进算法建议（例如，使用筛法替代试除法）。
如果代码在算法层面已经达到最优，才能回答“无需改进”。

请直接输出你的反馈，不要包含任何额外的解释。
"""

# =================优化=================
REFINE_PROMPT_TEMPLATE = """
你是一位资深的Python程序员。你正在根据一位代码评审专家的反馈来优化你的代码。

# 原始任务:
{task}

# 你上一轮尝试的代码:
{last_code_attempt}

评审员的反馈：
{feedback}

请根据评审员的反馈，生成一个优化后的新版本代码。
你的代码必须包含完整的函数签名、文档字符串，并遵循PEP 8编码规范。
请直接输出优化后的代码，不要包含任何额外的解释。
"""
````

❓ 我的疑问

这里的代码范例中给出了一个`get_trajectory()`的代码实现，用来总结过往迭代的所有执行结果和反馈，也即长期记忆，但后续并没有实际调用。

</div>
    
对于以上问题，可以在REFINE_PROMPT_TEMPLATE中加入一个对应模块
```
# 完整的历史轨迹:
{trajectory}
```
随后在实际调用中将`last_code`替换为完整轨迹：

```python!
refine_prompt = REFINE_PROMPT_TEMPLATE.format(
    task=task,
    trajectory=self.memory.get_trajectory(),  # 使用完整轨迹，替换last_code_attempt=last_code,
    feedback=feedback
)
```

这样在做优化的时候，就可以考虑长期记忆进行对应优化。

<div class="note insight">

💡 我的想法
可以看到，reflection主要针对的反思角度是什么是由prompt决定的，这里设定的反思模块主要针对效率进行分析，因而prompt在设计的时候就要针对业务的最终目标。

</div>

### 风险与收益

风险：
- 开销变大，相当于每个轮次调用三次llm
- 延迟变大，因为三个llm需要串行执行
- 维护与调试困难，prompt写三份

收益：
+ 解决的质量提升
+ 鲁棒性增强

## 总结

<div class="note insight">

💡 我的思考 三者的本质区别
实际上都有思考机制，但是思考发生的时间与方式不同。
    
react每次行动后做一次观察和思考，相当于online。但是每次的思考只为当前步骤最优，但不考虑当前这步会不会让下一步更难

plan的问题在于plan是独立形成的，没有基于任何动作的反馈。后续执行时可能与plan偏离。

reflection相当于high level的react，但他是对整个trajectory做优化，相当于考虑全局最优。可以根据经验推倒重来，react不能回头。

是不是可以三者融合，plan制定计划，react每步最优，reflection在一个轮次后整体调整。

</div>
