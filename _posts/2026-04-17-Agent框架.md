## 前言

过去的几篇文章中手搓了几种不同的agent，在实现的过程中逐渐发现有很多地方其实是可以进行复用的，比如底层的LLM模块（当然这一块已经复用了），类似的tool calling，甚至是将agent封装后在更复杂的多agent结构中进行调用。

那么，有没有什么办法可以避免这种重复的劳动呢？噔噔噔！Agent框架来也。

本篇主要为基于[HelloAgent第六章](https://datawhalechina.github.io/hello-agents/#/./chapter6/%E7%AC%AC%E5%85%AD%E7%AB%A0%20%E6%A1%86%E6%9E%B6%E5%BC%80%E5%8F%91%E5%AE%9E%E8%B7%B5)的学习笔记以及个人感悟。

## 什么是Agent框架

在开始之前，先要搞明白的问题是：什么是Agnet框架？框架的本质是对围绕agent的共性的、常复用的功能的抽象和封装。他将不同agent会用到的模块打包封装起来，比如tool、memory、model，要用的时候直接像拼积木一样从中选出需要的模块进行快速的组装。

> 和深度学习中的PyTorch或tensor类似，将不同种类的神经网络提前封装好，写代码的时候只需要进行组装，而不用一个个去实现不同网络的back propagation。

具体来说，agent框架将带来以下好处：

1. **提升代码复用和开发效率**。很好理解，避免重复造轮子。
2. **提升agent结构的可扩展性**。不同的模块被提前定义好，之间的接口进行统一，在搭建的时候可以很方便的进行结构的改变和扩展。
3. **更强大的状态管理**。在Reflection中实现的memory模块即为一个非常初期的状态管理，在实际的工程中状态管理将更加困难，也需要更多的优化，以为会涉及更长的上下文、更复杂的工具调用、跨轮次的对话记忆等等方面。
4. **方便debug**。可以通过hook机制在固定节点打印日志，而无需开发者手动频繁打印状态信息。

接下来看几个主流的agent框架，并最终实现一个自己的简单agent框架。

## AutoGen

### 总体架构

框架分为三个核心部分：
1. Core，封装了与LLM的底层直接交互部分。
2. AgentChat，在core之上提供的高级接口
3. Extensions，所有的扩展功能，包括用户可自定义的部分

此外AutoGen的一大特点是实现了**异步**编程。在一般的agent实现流程中，等待llm的反馈通常是较为耗时的部分，容易造成流程阻塞。用异步模式在等待llm相应的过程中进行其他可同步执行的任务，提高并发处理能力。

### 核心智能体组件

智能体是最核心的组件，被分为了两类：

* AssistantAgent：本质上是封装了一层LLM，主要负责思考。根据不同的预设prompt可以被赋予不同的角色定位，如developer，committer等等，完成不同的任务，但通常只思考，不调用外部工具，输入也只是其他agent的输出消息。
* UserProxyAgent：本质依然是封装LLM，不同在于能力与定位上。其具有与外部交互的能力，会被作为执行器调用外部工具获取额外的信息。更特殊的，会被视为人类用户的“使者”。总之是与“外部”交互的窗口

### 多Agent协调机制

当同时管理多个agent时，必定需要一种机制来协调与不同agent以及不同agent之间的对话。AutoGen使用了[Teams](* SelectorGroupChat：)的概念，也即群聊，并设定了不同的沟通机制，比如：
* RoundRobinGroupChat：简单的循环机制，按设定好的顺序轮流参与。适用于流程较为明确的场景，比如软件开发流程，文档-评审-代码-评审。在Reflection中使用的本质也是类似的循环机制。
* SelectorGroupChat：会有一个“主理人”agent，负责在每个agent执行完后指定下一位。
* Swarm：每个agent执行完后，发回的HandoffMessage中除了返回的内容还会指定发送给谁，也就是下一个speaker。

### 代码demo：开发团队

通过AutoGen实现对一个开发团队的完整复刻，不同agent将在其中充当软开团队中的不同角色。定义了四种不同的角色：PM，开发，committer以及用户，其中前三种使用AssistantAgent，最后用户使用UserProxyAgent。

#### 角色定义

以PM为例，其定义如下：

```python
def create_product_manager(model_client):
```

而对于用户智能体则略有不同，其代码如下：

#### 沟通机制

一个软件开发的流程是相对固定的，因此可以选用RoundRobinGroupChat的轮询机制，按照业务逻辑“需求-编码-review-测试”的流程，编排agent顺序。

```python
# 定义团队聊天和协作规则
team_chat = RoundRobinGroupChat(
    participants=[
        product_manager,
        engineer,
        code_reviewer,
        user_proxy
    ],
    termination_condition=TextMentionTermination("TERMINATE"),
    max_turns=20,
)
```

* participants定义了不同的参与agent，也一并确定了顺序
* termination_condition定义了终止条件，这里也就是当TERMINATE信号发出
* max_turns防止无限循环

#### 启动循环

由于最新的AutoGen架构采用异步机制，整个流程都运行在一个异步函数中，最终通过`asyncio.run()`执行。

```python
async def run_software_development_team():
    # ... 初始化客户端和智能体 ...
    
    # 定义任务描述
    task = """我们需要开发一个比特币价格显示应用，具体要求如下：
            核心功能：
            - 实时显示比特币当前价格（USD）
            - 显示24小时价格变化趋势（涨跌幅和涨跌额）
            - 提供价格刷新功能

            技术要求：
            - 使用 Streamlit 框架创建 Web 应用
            - 界面简洁美观，用户友好
            - 添加适当的错误处理和加载状态

            请团队协作完成这个任务，从需求分析到最终实现。"""
    
    # 异步执行团队协作，并流式输出对话过程
    result = await Console(team_chat.run_stream(task=task))
    return result

# 主程序入口
if __name__ == "__main__":
    result = asyncio.run(run_software_development_team())

```

这里的重点是对异步执行的理解，因为之前对python内的异步执行了解不多，这里着重查理解一下。

在一个线程中，CPU执行代码的速度极快，然而，一旦遇到IO操作，如读写文件、发送网络数据时，就需要等待IO操作完成，才能继续进行下一步操作。这种情况称为同步IO。因为一个IO操作就阻塞了当前线程，导致其他代码无法执行，所以我们可以使用多线程或者多进程来并发执行代码，为多个用户服务。

多线程和多进程的模型虽然解决了并发问题，但是系统不能无上限地增加线程。由于系统切换线程的开销也很大，所以，一旦线程数量过多，CPU的时间就花在线程切换上了，真正运行代码的时间就少了，结果导致性能严重下降。

另一种解决IO问题的方法是异步IO。当代码需要执行一个耗时的IO操作时，它只发出IO指令，并不等待IO结果，然后就去执行其他代码了。一段时间后，当IO返回结果时，再通知CPU进行处理。

具体到我们的代码中，先看最外层，`asyncio.run(run_software_development_team())`会启动一个python事件循环，`把run_software_development_team()`这个异步函数跑起来，并进行管理。

函数内部的关键则在于
```python
result = await Console(team_chat.run_stream(task=task))
```
拆开来看，`team_chat.run_stream(task=task)`开始多轮对话，并且流式输出内容，也就是一边运行一边输出。`Console`负责打印到终端。执行到`await Console(team_chat.run_stream(...))`时，如果底层在等网络响应，当前协程会“让出控制权”，去执行整个事件循环中可执行的下一件事。

需要注意的一点是，不同agent之间还是以绝对串行的方式在执行。

### 总结

整个流程中我们不再像之前那样需要手动拼凑所有的context，几个agent之间的信息传递对用户来说是不可见的。但这也同样带来了问题，就是调试的难度有所增加。

终止条件目前用的是用户输入特定字符，但整个框架提供了更多选项，比如最大迭代、外部函数、甚至根据token数量终止。这就是框架带来的优势，所有的东西都在那里，要用哪个拿哪个。

## AgentScope

### 架构特性

由阿里达摩院开发，将记忆、工具、LLM、message封装在foundation层，内置了不同种类的agent并提供了ReAct的实现。最重要的是，提供了多agent的合作方式，如MsgHub和Pipeline。

在AgentScope中，所有agent之间的交互被抽象为message，而非传统的函数调用。这样做的好处是：
1. 异步解耦：agent之间只管发送消息而无需等待对方回复，一个输入处理完直接处理下一条
2. 调试性更好：message均可被记录，且由于包含更多meta信息如发送者、事件等，可以在事后复盘发送关系，便于调试和监控

整个消息的处理中心是MsgHub，有以下特点：
1. 灵活的消息路由：可以选择不同的消息处理方式，点对点、广播、组播等
2. 消息持久化：可以将消息进行记录，如上所诉

所有的agent均基于基类AgentBase进行开发，只需要着重关心核心的reply方法即可，其余部分不需要过分关注.

### 代码实现

文中创建了一个狼人杀项目，通过不同角色势力之间的沟通来展现MsgHub对于不同message分发方式的实现。整体的项目就不看了，只看一些片段来了解部分作用机理。以狼人部分为例：
```python
async def werewolf_phase(self, round_num: int):
    """狼人阶段 - 展示消息驱动的协作模式"""
    if not self.werewolves:
        return None
        
    # 通过消息中心建立狼人专属通信频道
    async with MsgHub(
        self.werewolves,
        enable_auto_broadcast=True,
        announcement=await self.moderator.announce(
            f"狼人们，请讨论今晚的击杀目标。存活玩家：{format_player_list(self.alive_players)}"
        ),
    ) as werewolves_hub:
        # 讨论阶段：狼人通过消息交换策略
        for _ in range(MAX_DISCUSSION_ROUND):
            for wolf in self.werewolves:
                await wolf(structured_model=DiscussionModelCN)
        
        # 投票阶段：收集并统计狼人的击杀决策
        werewolves_hub.set_auto_broadcast(False)
        kill_votes = await fanout_pipeline(
            self.werewolves,
            msg=await self.moderator.announce("请选择击杀目标"),
            structured_model=WerewolfKillModelCN,
            enable_gather=False,
        )

```
通过`async with MsgHub(...) as werewolves_hub`创建专属于狼人角色的异步通信通道，其中第一项`self.werewolves`本质是一个ReactAgent的list，是这个MsgHub的所有参与者，也就是只有狼人角色可以看到。并且开启了广播`enable_auto_broadcast=True`，一个狼人发送msg所有狼人角色收到。`announcement`为初次公告，这里异步调用主持人的公告函数发表声明。

```python
for _ in range(MAX_DISCUSSION_ROUND):
    for wolf in self.werewolves:
        await wolf(structured_model=DiscussionModelCN)
```
这一段中所有狼人根据主持人的消息，开始异步发送消息，每个后续的狼人根据消息池中的当前状态发送代表自己思考的消息。

后续的`kill_votes = await fanout_pipeline()`给所有狼人发送同样消息，收集所有狼人的投票message，但狼人之间的消息互不可见。

并且，既然是游戏，就有一定要遵守的规则，比如狼人只能刀一个人，女巫不能同守同救。一般这样的规则需要手动在prompt中进行约束，非常繁琐。AgentScope提供了结构化的输出机制，很好的规范了这一点：

```python
class DiscussionModelCN(BaseModel):
    """讨论阶段的输出格式"""
    reach_agreement: bool = Field(
        description="是否已达成一致意见",
        default=False
    )
    confidence_level: int = Field(
        description="对当前推理的信心程度(1-10)",
        ge=1, le=10,
        default=5
    )
    key_evidence: Optional[str] = Field(
        description="支持你观点的关键证据",
        default=None
    )

```

这样会自动在prompt中生成类似于：

```markdown
请根据当前讨论内容，输出如下JSON格式：

{
  "reach_agreement": true/false, #是否已达成一致意见
  "confidence_level": 1-10, #对当前推理的信心程度(1-10)
  "key_evidence": "...", # 支持你观点的关键证据
}
```

llm便可以进行对应的格式化输出，避免了繁琐的自己构造prompt的流程。


### 总结

将agent之间的交流抽象为msg的想法很好很直观，并且自然地产生了不同的信息交互方式，很适合多agent的场景使用。规范化输出格式，让需要遵循特定规则的场景更容易将规则书面化。

## LangGraph

### 结构

LangGraph是LangChain的扩展，将agent的执行流程建模为状态机，由state、node、edge共同定义整个网络。首先来看着三个重要元素

#### State

是一种全局状态，所有节点均可读取和更新这个状态，通常定义如下：
```python
class AgentState(TypedDict):
    messages: List[str]      # 对话历史
    current_task: str        # 当前任务
    final_answer: str        # 最终答案
    # ... 任何其他需要追踪的状态
```

#### Node

每个节点都是一个接收**当前状态**作为输入、并返回一个**更新后的状态**作为输出的 Python 函数。节点是执行具体工作的单元。比如可以定义一个执行器节点：
```python
# 定义一个“执行者”节点函数
def executor_node(state: AgentState) -> AgentState:
    """执行最新计划，并更新状态。"""
    latest_plan = state["messages"][-1]
    # ... 执行计划并获得结果 ...
    result = f"执行计划 '{latest_plan}' 的结果..."
    
    state["messages"].append(result)
    return state
```

#### Edge

负责节点间的信息传递，类似于AgentScope里message的概念。最简单的是常规边，由一个节点输出流向另一个固定节点。特别的是条件边，通过函数根据当前状态来判断流向的下一个节点是哪里。例如：

```python
def should_continue(state: AgentState) -> str:
    """条件函数：根据状态决定下一步路由。"""
    # 假设如果消息少于3条，则需要继续规划
    if len(state["messages"]) < 3:
        # 返回的字符串需要与添加条件边时定义的键匹配
        return "continue_to_planner"
    else:
        state["final_answer"] = state["messages"][-1]
        return "end_workflow"
```

有了这三个，就可以组成最终的graph。接下来以一个简单的项目看看整体的工作流程。

### 问答助手demo

这里我们通过LangGraph来实现一个简单的问答助手，分别定义全局状态，不同作用的节点，再将之连接成图。

#### 全局状态
首先我们定义全局状态如下：

```python
class SearchState(TypedDict):
    messages: Annotated[list, add_messages]
    user_query: str      # 经过LLM理解后的用户需求总结
    search_query: str    # 优化后用于API的搜索查询
    search_results: str  # 搜索返回的结果
    final_answer: str    # 最终生成的答案
    step: str            # 标记当前步骤
```

这里的user_query并非直接的用户输入prompt，而是大模型提炼后的关键词list，之后放入search_query中供后续api调用。

#### 节点
我们需要以下三个步骤：
1. 理解：分析用户意图
2. 搜索：搜索相关信息
3. 回答：汇总信息生成答案

接下来根据以上三个步骤定义三种不同的节点，首先是理解节点：

```python
def understand_query_node(state: SearchState) -> dict:
    """步骤1：理解用户查询并生成搜索关键词"""
    user_message = state["messages"][-1].content
    
    understand_prompt = f"""分析用户的查询："{user_message}"
    请完成两个任务：
    1. 简洁总结用户想要了解什么
    2. 生成最适合搜索引擎的关键词（中英文均可，要精准）

    格式：
    理解：[用户需求总结]
    搜索词：[最佳搜索关键词]"""

    response = llm.invoke([SystemMessage(content=understand_prompt)])
    response_text = response.content
    
    # 解析LLM的输出，提取搜索关键词
    search_query = user_message # 默认使用原始查询
    if "搜索词：" in response_text:
        search_query = response_text.split("搜索词：")[1].strip()
    
    return {
        "user_query": response_text,
        "search_query": search_query,
        "step": "understood",
        "messages": [AIMessage(content=f"我将为您搜索：{search_query}")]
    }
```
通过state中的messages字段，过滤HumanMessage类型的消息来获取用户输入的原始问题。通过prompt引导llm总结用户的需求，然后限定llm的格式化输出，从中提取关键词。多了一步引导，类似COT的效果让结果更优。返回更新后的state。

搜索节点通过state获取到搜索的关键词，调用api进行搜索，主要是api相关解析。

回答节点将用户原始问题和搜索结果植入prompt中，进行总结回答。得到的结果再次用于更新state：
```python
def generate_answer_node(state: SearchState) -> dict:
    """步骤3：基于搜索结果生成最终答案"""
    if state["step"] == "search_failed":
        # 如果搜索失败，执行回退策略，基于LLM自身知识回答
        fallback_prompt = f"搜索API暂时不可用，请基于您的知识回答用户的问题：\n用户问题：{state['user_query']}"
        response = llm.invoke([SystemMessage(content=fallback_prompt)])
    else:
        # 搜索成功，基于搜索结果生成答案
        answer_prompt = f"""基于以下搜索结果为用户提供完整、准确的答案：
            用户问题：{state['user_query']}
            搜索结果：\n{state['search_results']}
            请综合搜索结果，提供准确、有用的回答..."""
        response = llm.invoke([SystemMessage(content=answer_prompt)])
    
    return {
        "final_answer": response.content,
        "step": "completed",
        "messages": [AIMessage(content=response.content)]
    }
```

#### 连接成图
最后将所有节点添加，并通过edge进行连接，即得到我们最终的结果：
```python
def create_search_assistant():
    workflow = StateGraph(SearchState)
    
    # 添加节点
    workflow.add_node("understand", understand_query_node)
    workflow.add_node("search", tavily_search_node)
    workflow.add_node("answer", generate_answer_node)
    
    # 设置线性流程
    workflow.add_edge(START, "understand")
    workflow.add_edge("understand", "search")
    workflow.add_edge("search", "answer")
    workflow.add_edge("answer", END)
    
    # 编译图
    memory = InMemorySaver()
    app = workflow.compile(checkpointer=memory)
    return app
```

> 最后这一步的感觉可太像组装pytorch的网络了

### 总结

LangGraph的定义非常的易懂，大概有一些最最基础图知识的人，几乎一下子就能明白这个框架的整体思想。LangGraph的模块化程度更高，不论是node还是edge都可以单独定义然后互相组装，更有利于快速创建不同的结构进行敏捷开发。并且天然的适配流程图，有流程图几乎就可以一比一的搭建出对应的graph

## 总结的总结

* AutoGen：对话驱动，不同的聊天方式决定了不同的agent之间协作方式
* AgentScope：agent之间的合作抽象为message，通过MsgHub以及message内容决定传递的方向，进一步定义agent之间的关系
* LangGraph：定义好节点内容后，图结构决定一切，通信全靠state

<div class="note thought" markdown="1">

💡 思考

三者在设计理念上的一个区别维度是信息的传递方向由什么决定，或者说整体的流程如何构建由什么决定，以及agent之间的信息是通过何种形式进行传递的。

对于AutoGen，其预定义好的“聊天方式”决定这一切，比如RoundRobinGroupChat定义的轮流发言，或者由主持人决定的下一个发言顺序，总体来说自由度偏低，无法真正全自由度的定义沟通方向。信息则是通过直接输出的内容进行传递，因为传递的方向与之解耦了。

AgentScope则是由MsgHub和message共同进行了定义，每一封message从谁发出来，发向哪里，谁可见，决定了整体的流程走向。基本可实现所有想要的流程方向，不过在定义流程的时候不是很直观，需要根据message内容脑子里有个结构。

LangGraph最直接，直接通过搭积木的方式构建所有节点和节点之间的关系，一目了然。信息传递则通过共享状态，可以通过特定筛选看到想看的信息，但有时会有些冗余。
