## 前言

最近在写框架，也去看了不少开源的框架项目，在其中经常能看到一些`async`的相关操作，之前没有了解过python中的异步执行，趁机学习了一下。

## 异步

首先，什么是异步？需要明确的是，异步是在同一个进程内的事情，与并发无关。恰恰相反的是，使用异步的场景通常是多个任务之间存在依赖关系，而其中一个或多个任务需要较长的时间来等待**外部**返回结果，如果不使用异步会造成阻塞。这个时候就会使用异步操作，将容易造成阻塞的任务挂起，先去执行与他没有依赖关系的任务，进而使得CPU的利用率保持在一个较高的水平上。


Python异步运行机制的核心是事件循环，它会不断从ready queue中取出可以立即执行的任务片段并运行。整个机制主要围绕两部分展开：

* ready queue：存放可以立刻继续执行的任务
* 等待中的任务：以Task或Future形式存在，等待外部条件完成

根据等待来源的不同，这些任务可以分为三类：

* 基础Future：普通异步任务，完成时间不确定
* 定时器（scheduled timers）：如sleep，在未来某个时间点触发
* I/O事件（I/O selector）：如socket或文件描述符的读写完成

在Python异步中，create_task()或gather()会将协程包装成Task并交给事件循环管理。事件循环维护ready queue，同时监听定时器和I/O事件，用于触发等待中的任务。

当任务执行到await时，如果等待对象尚未完成，该任务会被挂起，并将“恢复执行”的逻辑注册到对应的Future或事件源上。事件循环不会阻塞，而是继续执行ready queue中的其他任务。

当被await的对象完成后，事件循环会将该任务的下一步执行重新加入ready queue，随后在调度过程中继续从await之后运行。


## 看看代码

用下面这个小demo，看一下实际运行中异步的效果，并通过一些内部变量的打印来看看实际的调度逻辑。

```python
async def fetch(name, delay):
    print(f"{name} 开始")
    await asyncio.sleep(delay)
    print(f"{name} 结束")
    return f"{name} done"

async def monitor_tasks(interval=0.2):
    while True:
        loop = asyncio.get_running_loop()
        tasks = asyncio.all_tasks()
        active_names = sorted(task.get_name() for task in tasks if not task.done())
        ready_size = len(list(getattr(loop, "_ready", [])))  # 调试用私有字段
        print(f"[monitor] 活跃任务({len(active_names)}): {active_names} | ready_queue={ready_size}")
        await asyncio.sleep(interval)

async def main():
    monitor = asyncio.create_task(monitor_tasks(), name="monitor")
    task_a = asyncio.create_task(fetch("任务A", 2), name="任务A")
    task_b = asyncio.create_task(fetch("任务B", 1), name="任务B")
    result1, result2 = await asyncio.gather(task_a, task_b)
    
if __name__ == "__main__":
    asyncio.run(main())
```

### asyncio.run(main())

`asyncio.run(main())` 可以理解为三步：

1. 创建并启动事件循环（event loop）；
2. 把 `main()` 这个协程放进去执行；
3. 等 `main()` 完成后，清理循环并退出。

注意这里的要求是`run(...)` 的参数必须是**协程对象**（通常来自 `async def` 函数调用）。

### create_task + gather

在 `main` 里：

- `asyncio.create_task(fetch("任务A", 2), name="任务A")`
- `asyncio.create_task(fetch("任务B", 1), name="任务B")`

这两句会把协程包装成 `Task` 并注册到事件循环，也就是让A/B任务进入调度。随后 `await asyncio.gather(task_a, task_b)` 的含义是**并发推进 A/B，直到二者都完成，再把结果按传入顺序返回。** `monitor`作为辅助理解的函数，也是类似的操作，但就不单独说明了。

### await与调度时间线
结合打印的结果来看，结果如下：

```
[monitor] 活跃任务(4): ['Task-1', 'monitor', '任务A', '任务B'] | ready_queue=2
任务A 开始
任务B 开始
[monitor] 活跃任务(4): ['Task-1', 'monitor', '任务A', '任务B'] | ready_queue=0
[monitor] 活跃任务(4): ['Task-1', 'monitor', '任务A', '任务B'] | ready_queue=0
[monitor] 活跃任务(4): ['Task-1', 'monitor', '任务A', '任务B'] | ready_queue=0
[monitor] 活跃任务(4): ['Task-1', 'monitor', '任务A', '任务B'] | ready_queue=0
任务B 结束
[monitor] 活跃任务(3): ['Task-1', 'monitor', '任务A'] | ready_queue=0
[monitor] 活跃任务(3): ['Task-1', 'monitor', '任务A'] | ready_queue=0
[monitor] 活跃任务(3): ['Task-1', 'monitor', '任务A'] | ready_queue=0
[monitor] 活跃任务(3): ['Task-1', 'monitor', '任务A'] | ready_queue=0
[monitor] 活跃任务(3): ['Task-1', 'monitor', '任务A'] | ready_queue=0
任务A 结束
任务A done 任务B done
总耗时: 2.00s
```
根据打印结果，实际调度的时间线如下：
- 第一行 `ready_queue=2`：A/B 刚创建，已进入可运行队列；
- A运行到 `await sleep(...)` 后会挂起，不再处于 ready queue，转而执行ready queue中的下一个任务，也就是B
- B也运行到`await`，执行相同操作
- 所以后续常看到 `ready_queue=0`，但不代表“没有任务”，只是它们在等待唤醒；
- B 先结束后，被放入ready queue，但因为时间很短所以我们的monitor没有捕捉到；下一轮循环被取出，执行后续的操作，瞬间完成，活跃任务从 4 变 3；
- A 结束后类似操作，几乎瞬间完成
- `gather` 返回，主流程打印结果。

## 总结
异步执行的核心是让注册的任务在发生阻塞时让出执行权，不断从可运行队列中取出下一个可执行任务运行，从而避免轮空等待。

擅长的是等待型场景，网络请求、磁盘、数据库、sleep 等。对于LLM来说，等待LLM返回答复就是很适合的应用场景，因此在agent框架中高频出现。
